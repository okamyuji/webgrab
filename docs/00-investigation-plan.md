# 調査計画書 — LLM Web検索の取得量制限の実測

- バージョン: 1.0
- 日付: 2026-07-17
- 対象の主張: 「Claude等のLLMのweb searchは先頭の約2000バイト（文字）しか取得せず残りを読み捨てている、もしくは要約だけを取得している」（Zenn記事由来、記事URLは未特定）

## 前提の仕分け

- 事実: Claude Codeには WebSearch（検索スニペット返却）と WebFetch（URL取得+小型モデルで要約応答）の2系統がある
- 仮定: 記事の主張はこの2系統のいずれか（または両方）の取得量制限を指している
- 不明: 記事の正確なURL・対象ツール（Claude web UI / API web search tool / Claude Code）。→ 本調査ではClaude Code環境で観測可能な範囲を計測し、断定はその範囲に限定する

## MECEロジックツリー（何を計測すれば主張の真偽が決まるか）

```text
主張「先頭2000バイトのみ/要約のみ」の検証
├── A. WebSearch（検索ツール）
│   ├── A1. 1結果あたりのスニペット長（文字数）の実測
│   └── A2. スニペットが原文抜粋か要約かの判定（原文との照合）
├── B. WebFetch（URL取得ツール）
│   ├── B1. 取得上限の実測（既知サイズ・マーカー入りページで可視範囲を特定）
│   │   ├── B1a. 先頭からの連続取得か（マーカーの連続性）
│   │   └── B1b. 上限バイト数/文字数の推定（見えた最終マーカー位置）
│   └── B2. 返却形式の判定（原文そのままか、小型モデルの要約か）
└── C. 対照実験（測定系の妥当性）
    ├── C1. マーカーページ自体の正しさ（curlで全マーカー存在を確認）
    └── C2. 小さいページ（上限未満）で全文が見えることの確認
```

## 計測手順

1. マーカー入りHTMLページを生成（500文字ごとに `MARKER-NNNNNN`（NNNNNN=文字オフセット）を埋め込み、総量約100,000文字、末尾に固有文）
2. ローカルHTTPサーバで配信し、WebFetchで「見えているMARKERをすべて列挙し、最後のMARKERと本文末尾の文を引用せよ」と指示
3. ローカル不可の場合の代替: Artifactとして公開しそのURLをWebFetchする
4. WebSearchは既知ページを検索し、返却スニペットの文字数を計測する
5. 結果を `01-measurement-report.md` に記録する

## 完了条件

- B1の上限値（またはマーカー全件可視＝上限なしの証拠）が数値で記録されている
- A1のスニペット長分布が数値で記録されている
- 主張の真偽判定（真/偽/部分的に真）が観測データ付きで下されている

## やらないこと

- Anthropicサーバ内部実装の断定（観測範囲外）
- ChatGPT/Kimi等の他社ツールの実測（本環境から観測不能。一般論としての文献調査のみ）
